\chapter{智能体编排系统概要设计}

\section{系统结构设计}

系统采用分层架构与前后端分离的部署模式\upcite{evans2016dddcn}，后端按照接口层、应用层、领域层、
基础设施层四层划分，前端基于 React 与 TypeScript 实现可视化画布与业务交互界面。
系统整体结构如图~\ref{fig:sys_struct}~所示。

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=0.88\textwidth]{figures/fig_arch.png}
  \caption{系统整体结构图}
  \label{fig:sys_struct}
\end{figure}

各层的核心职责如下：

接口层：对外提供 RESTful 接口与实时推送通道，负责处理 HTTP 请求、
参数校验以及数据传输对象与领域对象之间的转换。用户注册、登录、
知识库管理、审批接口与工作流启动接口均在此层对外暴露。

应用层：包含 Agent 管理服务、工作流调度服务、知识应用服务与用户应用服务等。
该层负责跨聚合的编排，例如协调 Agent 查询、工作流图结构解析、知识检索增强、
审批恢复与异步文档处理等，但不承载具体的领域规则本身。

领域层：系统业务逻辑的核心，包含工作流执行聚合、工作流图结构、知识文档
与用户等核心领域对象，并定义了向量检索、文件存储、文本切分、邮件服务等端口接口。
领域层保持纯净，不依赖任何框架或外部库。

基础设施层：实现领域层定义的端口接口，包括节点执行策略、关系型数据库仓储适配、
分布式缓存客户端、向量数据库适配、对象存储适配、邮件发送实现以及大模型框架适配。

前端基于 React 与 TypeScript 实现，包含 Agent 列表与画布编排页、对话执行页、
人工审批页、知识库页与登录注册页等功能页面，并通过实时推送通道接收工作流执行事件。

\subsection{用户认证与系统安全支撑模块}

该模块为整个平台提供统一的身份识别、访问控制与安全约束能力。系统支持邮箱注册、
登录、找回密码与访问令牌认证，其中验证码发送、密码强度校验、密码强哈希存储与登录失败限制
均在认证领域服务中实现。认证成功后，接口层在请求上下文中解析用户身份，业务模块
据此执行 Agent 所有权校验、知识库访问控制、审批操作记录等逻辑。

在数据流层面，用户注册或登录成功后系统签发访问令牌，前端在后续请求中统一携带；
接口层解析令牌并写入当前用户上下文，应用服务据此进行资源所有权校验：例如 Agent 更新需校验是否属于当前用户、
知识文档上传需校验数据集归属、人工审批记录需保存当前审批人标识。因此，用户认证与系统安全
并非孤立模块，而是贯穿其他四个业务模块的基础支撑层。

\subsection{Agent 管理与可视化编排模块}

该模块负责 Agent 的全生命周期管理与可视化流程定义。后端的 Agent 应用服务
实现创建、编辑、发布、版本查询与回滚等能力，每个 Agent 对应一份以图结构
形式存储的工作流定义；前端通过 ReactFlow 画布提供拖拽式编排界面，支持节点
创建、删除、参数配置与边连接，并将画布状态统一序列化为图结构数据。
在发布流程中，系统会将当前草稿快照写入版本表并更新 Agent 的已发布版本号，
保证开发中的草稿与线上执行版本相互隔离。

在数据流层面，用户在前端画布完成 Agent 编排后，节点与边被序列化为图结构数据；
保存草稿时由后端校验图结构合法性，发布时生成版本快照并固化为可执行版本，
运行时调度器再依据 Agent 标识与版本号读取对应的图定义作为工作流执行的输入，
形成“画布编排 → 草稿保存 → 版本发布 → 运行时执行”的完整链路。

\subsection{工作流调度执行模块}

工作流执行模块以调度服务与工作流执行聚合的协作为核心。系统在执行开始时读取
Agent 的图定义，将其反序列化为内存中的工作流图结构，随后初始化执行上下文、节点
状态与就绪队列。执行引擎基于图结构的拓扑顺序与事件驱动调度，支持无依赖节点的
并行执行、条件分支、路径剪枝以及执行状态的整体推进。整个执行过程通过实时推送通道
持续向前端发送节点启动、完成、错误以及暂停恢复等事件。

\subsection{人工审核与恢复机制模块}

该模块在工作流执行链路中植入“暂停 — 等待 — 恢复 / 终止”的控制能力。
节点可配置人工检查点的开关与触发阶段，支持在节点执行前或执行后两种时机暂停工作流。
设计上，系统采用分布式缓存维护待审批队列，并为每个执行实例配置分布式锁；
审批人通过审批列表接口查看待处理任务，进入详情后可查看当前节点的上下文、
修改中间结果并提交通过或拒绝操作。恢复或拒绝请求均以执行版本号作为并发控制条件，
从而避免多人并发审批时出现状态覆盖问题。

\subsection{知识库与检索增强模块}

该模块为工作流中的知识节点与大模型节点提供外部知识支撑。从设计上看，知识链路
由数据集管理、文档管理、异步处理与语义检索四部分组成。用户上传文档后，系统先将
原始文件存入对象存储，再由异步处理器完成文档解析、文本分块与向量化入库；向量数据与
原始文档元信息通过文档标识、数据集标识等关联字段建立联系，便于后续的删除、筛选与语义搜索。

在数据流层面，文档从上传到可检索的链路为：接口层校验文件 → 应用层创建文档记录并触发
异步处理 → 基础设施层完成解析、分块、嵌入向量并写入向量数据库；运行时检索链路为：
工作流知识节点拼接查询 → 嵌入模型生成查询向量 → 向量数据库语义检索 → 召回片段注入大模型上下文。
这一双向链路保证了知识接入与运行时检索形成完整的闭环。

为支撑上述系统结构与功能模块的实现，本系统在后端运行环境、AI 应用框架、
数据存储、知识检索与前端等关键维度上进行了技术选型，
主要选型与依据如表~\ref{tab:tech_stack}~所示。

    \midrule
