\section{用户认证与系统安全支撑实现}

\subsection{注册、登录与密码找回流程}


用户认证模块是系统对外开放访问的基础支撑，主要围绕邮箱验证码、注册、登录与找回密码形成闭环。
该模块前端对应登录注册页面，后端由认证接口、用户应用服务、用户领域服务、验证码仓储和邮件服务共同完成。
实现过程中，前端负责表单输入、格式校验、按钮状态控制和错误提示；后端负责验证码生成、密码强度校验、用户唯一性校验、密码哈希存储和登录失败限流。
系统在验证码发送前校验邮箱格式和发送频率，验证码写入缓存后通过邮件服务发送给用户。

注册时，用户先输入邮箱并获取验证码，再填写用户名、密码和确认密码。
后端依次校验验证码、密码强度以及邮箱和用户名唯一性，校验通过后使用 BCrypt 保存密码哈希。
登录时，系统根据邮箱查询用户并校验密码，失败时记录限流信息，成功后清理失败计数并交由接口层签发令牌。
找回密码流程复用验证码校验机制，在新密码符合强度要求后完成密码重置。
该实现将“验证码有效性、密码安全性、登录限流和令牌签发”拆分到不同职责对象中，使认证流程既能支撑页面交互，也便于后续扩展短信验证码或第三方登录。

\begin{lstlisting}[style=java, caption={UserAuthenticationDomainService 用户认证核心逻辑}]
public void sendVerificationCode(String emailStr) {
    Email email = Email.of(emailStr);
    validateEmailSendRateLimit(email);
    String code = generateSecureCode();
    verificationCodeRepository.save(email.getValue(), code, 5);
    emailService.sendVerificationCode(email, code);
}

public User register(String emailStr, String code, String password, String username) {
    Email email = Email.of(emailStr);
    validateVerificationCode(email, code);
    validatePasswordStrength(password);
    User user = User.create(email, Username.of(username), BCryptUtil.encode(password));
    userRepository.save(user);
    verificationCodeRepository.delete(email.getValue());
    return user;
}

public User login(String emailStr, String password, String ip) {
    checkLoginRateLimit(emailStr, ip);
    User user = userRepository.findByEmail(Email.of(emailStr)).orElseThrow();
    if (!BCryptUtil.matches(password, user.getPasswordHash())) {
        recordLoginFailure(emailStr, ip);
        throw new IllegalArgumentException("密码错误");
    }
    clearLoginFailures(emailStr, ip);
    return user;
}
\end{lstlisting}

\section{Agent 管理与可视化编排实现}

\subsection{Agent 创建与版本管理}


Agent 管理模块承担工作流定义的生命周期管理职责，包括创建、修改、发布、回滚与版本查询。
该模块的实现目标不是单纯保存一个 Agent 名称，而是将前端可视化画布中的节点、边、节点参数和人工审核配置统一组织为可执行的图定义。
用户创建 Agent 时，系统根据默认模板生成初始 \texttt{graphJson}；用户在 React Flow 画布中完成节点拖拽、参数配置和连线后，前端将新的图结构提交到后端保存为草稿。

实现过程中，草稿保存只更新 Agent 当前编辑态配置，不影响已经发布的可执行版本；
发布操作会结合图校验器检查节点连通性、环路和必要参数，然后生成 \texttt{AgentVersion} 快照并更新当前已发布版本指针。
若后续需要恢复历史配置，系统可根据版本号读取快照并回滚当前 Agent。
该机制形成了"草稿编辑 → 发布快照 → 历史回滚"的版本管理链路，保证运行时图定义具备可追溯性，也避免用户正在编辑的草稿影响线上执行。

Agent 发布版本快照时，系统基于当前草稿构造版本快照并更新已发布指针，其核心逻辑如下。

\begin{lstlisting}[style=java, caption={AgentApplicationService Agent 发布关键代码}]
@Transactional(rollbackFor = Exception.class)
public void publishAgent(PublishAgentCmd cmd) {
    Agent agent = agentRepository.findById(cmd.getId()).orElseThrow();
    Integer nextVersion = agentRepository.findMaxVersion(agent.getId()).orElse(0) + 1;
    AgentVersion version = agent.publish(graphValidator, nextVersion);
    agentRepository.saveVersion(version);
    agent.setPublishedVersionId(version.getVersion().longValue());
    agentRepository.save(agent);
}
\end{lstlisting}

\subsection{工作流图定义与运行时衔接}


Agent 可视化编排的最终结果并不是静态页面数据，而是运行时调度的直接输入。
当前端保存画布后，节点、边、节点参数等结构统一写入 \texttt{graphJson} 字段。
该字段既作为编辑态草稿保存，也可在发布后固化为版本快照。

工作流执行启动时，调度器首先根据 \texttt{agentId} 查询 Agent 实体。
若用户显式指定了 \texttt{versionId}，则优先读取对应 \texttt{AgentVersion} 的图定义；
否则优先加载已发布版本，若不存在再回退到当前草稿。
随后，调度器通过图工厂将 \texttt{graphJson} 反序列化为领域层的
\texttt{WorkflowGraph}，供后续 \texttt{Execution} 聚合根执行调度。

这一机制实现了可视化编排与运行时执行的直接衔接，使前端画布不只是配置界面，
而是工作流引擎的图定义来源。由于执行时优先读取已发布版本，用户在画布中继续修改草稿不会影响已经启动的执行任务，从而保证配置迭代和流程运行之间的隔离。


调度器读取版本图定义并反序列化为工作流图的关键过程如下。

\begin{lstlisting}[style=java, caption={SchedulerService graphJson 装载与工作流图解析}]
public void startExecution(String executionId, Long agentId, Long userId,
        String conversationId, Integer versionId,
        Map<String, Object> inputs, ExecutionMode mode) {
    Agent agent = agentRepository.findById(agentId).orElseThrow();

    String graphJson;
    if (versionId != null) {
        graphJson = agentRepository.findVersion(agentId, versionId)
            .orElseThrow().getGraphJson();
    } else if (agent.getPublishedVersionId() != null) {
        graphJson = agentRepository.findVersion(agentId,
            agent.getPublishedVersionId().intValue())
            .orElseThrow().getGraphJson();
    } else {
        graphJson = agent.getGraphJson();
    }

    WorkflowGraph graph = workflowGraphFactory.fromJson(graphJson);
    Execution execution = Execution.create(executionId, agentId, userId,
        conversationId, graph, mode);
    startExecution(execution, inputs, mode);
}
\end{lstlisting}

\subsection{工作流执行启动模块}

\subsection{DAG 调度核心模块}


\texttt{Execution} 是工作流领域的聚合根，负责管理节点状态、执行上下文和整体状态流转。
该模块的实现重点在于把“节点完成事件”转化为“后续可执行节点集合”，并在同一位置处理条件分支、失败传播、人工暂停和整体完成等状态变化。
每个节点完成后，调度器调用 \texttt{advance(nodeId, result)} 写入节点状态与输出，
并根据结果判断是否需要条件分支剪枝、人工检查点暂停或继续推进后继节点。

在计算后继节点时，\texttt{getReadyNodes()} 会结合有效入度判断可运行节点；
已完成、已跳过或已失败的节点不再阻塞其后继节点，因此系统可以自动调度所有依赖已满足的节点并行执行。


\texttt{Execution.advance()} 的完整逻辑如下。

\begin{lstlisting}[style=java, caption={Execution.advance() DAG 调度推进关键代码}]
public List<Node> advance(String nodeId, NodeExecutionResult result) {
    nodeStatuses.put(nodeId, result.getStatus());
    if (result.getOutputs() != null) {
        context.setNodeOutput(nodeId, result.getOutputs());
    }
    if (result.isRouting()) {
        pruneUnselectedBranches(nodeId, result.getSelectedBranchId());
    }
    if (result.isPaused()) {
        this.status = ExecutionStatus.PAUSED_FOR_REVIEW;
        this.pausedNodeId = nodeId;
        this.pausedPhase = result.getTriggerPhase();
        this.version++;
        return Collections.emptyList();
    }
    this.version++;
    if (isCompleted()) {
        this.status = ExecutionStatus.SUCCEEDED;
        return Collections.emptyList();
    }
    if (hasFailed()) {
        this.status = ExecutionStatus.FAILED;
        return Collections.emptyList();
    }

    return getReadyNodes();
}
\end{lstlisting}

条件分支剪枝的递归逻辑如下。

\begin{lstlisting}[style=java, caption={Execution.skipNodeRecursively() 条件分支路径剪枝}]
private void skipNodeRecursively(String nodeId) {
    if (nodeStatuses.get(nodeId) != ExecutionStatus.PENDING) { return; }
    nodeStatuses.put(nodeId, ExecutionStatus.SKIPPED);

    Set<Node> successors = graph.getSuccessors(nodeId);
    for (Node successor : successors) {
        Set<Node> predecessors = graph.getPredecessors(successor.getNodeId());
        if (predecessors.size() <= 1) {
            skipNodeRecursively(successor.getNodeId());
        } else {
            boolean allSkipped = predecessors.stream().allMatch(
                pred -> nodeStatuses.get(pred.getNodeId())
                    == ExecutionStatus.SKIPPED);
            if (allSkipped) {
                skipNodeRecursively(successor.getNodeId());
            }
        }
    }
}
\end{lstlisting}

\begin{lstlisting}[style=java, caption={SchedulerService.checkPause() 人工检查点触发判断}]
private boolean checkPause(String executionId, Node node,
        TriggerPhase phase, StreamPublisher publisher,
        Map<String, Object> outputs) {
    HumanReviewConfig config = node.getHumanReviewConfig();
    if (config == null || !config.isEnabled()) { return false; }
    if (config.getTriggerPhase() != phase) { return false; }

    Execution execution = executionRepository.findById(executionId).orElseThrow();
    if (execution.isNodeReviewed(node.getNodeId())) { return false; }

    RLock lock = redisService.getLock("lock:exec:" + executionId);
    try {
        lock.lock(30, TimeUnit.SECONDS);
        execution.advance(node.getNodeId(), NodeExecutionResult.paused(phase, outputs));
        executionRepository.update(execution);
        humanReviewQueuePort.addToPendingQueue(executionId);
        publisher.publishEvent("workflow_paused", Map.of(...));
    } finally {
        if (lock.isHeldByCurrentThread()) { lock.unlock(); }
    }
    return true;
}
\end{lstlisting}

\subsection{工作流恢复模块}


审批通过后，前端调用 \texttt{POST /api/workflow/reviews/resume} 接口并携带执行编号、暂停节点、
乐观锁版本号以及可选编辑内容。服务端在分布式锁内校验暂停节点和版本号，
防止重复审批或并发恢复导致状态不一致。

实现过程中，系统会根据暂停阶段合并审批人修改的数据，写入审核记录，
调用 \texttt{execution.resume()} 将工作流恢复为运行态，并从 Redis 待审批队列移除该执行。
若暂停发生在节点执行后，调度器直接推进后继节点；若暂停发生在节点执行前，则重新调度当前节点执行。


\begin{lstlisting}[style=java, caption={SchedulerService.resumeExecution() 工作流恢复关键代码}]
public void resumeExecution(String executionId, String nodeId, Integer expectedVersion,
        Map<String, Object> edits, Long reviewerId, String comment) {
    RLock lock = redisService.getLock("lock:exec:" + executionId);
    try {
        lock.lock(30, TimeUnit.SECONDS);
        Execution execution = executionRepository.findById(executionId).orElseThrow();
        if (!pausedNode(execution, nodeId)) { return; }
        if (expectedVersion != null && !expectedVersion.equals(execution.getVersion())) {
            throw new OptimisticLockingFailureException("版本冲突");
        }
        TriggerPhase phase = defaultPhase(execution.getPausedPhase());
        mergeReviewEdits(execution, nodeId, edits, phase);
        humanReviewRepository.save(buildApproveRecord(executionId, nodeId, reviewerId, phase, comment));

        List<Node> readyNodes = execution.resume(nodeId, edits);
        executionRepository.update(execution);
        humanReviewQueuePort.removeFromPendingQueue(executionId);
        publishAndSchedule(executionId, nodeId, readyNodes, phase);
    } finally {
        if (lock.isHeldByCurrentThread()) { lock.unlock(); }
    }
}
\end{lstlisting}

\section{系统开发难点与解决方案}

在系统实现过程中，工作流调度、并发控制、实时推流等模块均遇到了若干工程难点。
本节梳理五个典型难点，阐述问题根因与对应解决方案，以体现系统在关键路径上的设计深度。

\subsection{DAG 环检测与死锁防护}


工作流引擎以 DAG 作为调度基础，如果用户在前端画布中无意创建了回路（如节点 A → B → C → A），
调度器在执行 \texttt{Execution.start()} 时就会陷入无限等待——所有节点相互依赖，
执行条件无法收敛，导致工作流永久挂起。


解决过程中，系统在 \texttt{start()} 调用时，在初始化节点状态之前优先调用工作流图的
\texttt{hasCycle()} 方法进行 DFS 递归栈判环：遍历每个节点时将当前访问链路记录到
\texttt{recursionStack}，若后继节点已经处于递归栈中，说明从当前路径回到了祖先节点，
图中存在循环依赖，系统立即抛出 \texttt{IllegalStateException}，并在接口层返回 HTTP 400。

\begin{lstlisting}[style=java, caption={WorkflowGraph.hasCycle() DFS 递归栈环检测}]
public boolean hasCycle() {
    Set<String> visited = new HashSet<>();
    Set<String> recursionStack = new HashSet<>();

    for (String nodeId : nodes.keySet()) {
        if (hasCycleDFS(nodeId, visited, recursionStack)) {
            return true;
        }
    }
    return false;
}

private boolean hasCycleDFS(String nodeId, Set<String> visited,
                            Set<String> recursionStack) {
    if (recursionStack.contains(nodeId)) {
        return true;
    }
    if (visited.contains(nodeId)) {
        return false;
    }

    visited.add(nodeId);
    recursionStack.add(nodeId);

    Set<String> successors = edges.getOrDefault(nodeId, Collections.emptySet());
    for (String successor : successors) {
        if (hasCycleDFS(successor, visited, recursionStack)) {
            return true;
        }
    }

    recursionStack.remove(nodeId);
    return false;
}
\end{lstlisting}

该检测在 \texttt{start()} 前执行，时间复杂度为 $O(V + E)$，对正常图无额外性能开销。

\subsection{人工审批并发一致性保障}


在多人协作场景下，同一暂停执行可能被多名审批人同时进入审批页面，
若没有任何保护机制，则可能出现：（a）两次审批成功导致执行状态被转换两次；
（b）先 approve 后 reject 导致结果不一致；（c）数据库更新覆盖彼此的版本号。


解决过程中，系统采用两层并发保护机制联合兜底：

第一层：Redis 分布式锁（Redisson RLock）。
所有 \texttt{pause / resume / reject} 操作均在锁 \texttt{lock:exec:\{executionId\}} 保护下执行，
超时设置为 30 秒，确保同一时刻只有一个线程进入临界区。

第二层：乐观锁（version 字段）。
\texttt{Execution} 聚合根维护 \texttt{version} 自增字段，前端发起审批时
携带当前 \texttt{expectedVersion}，服务端校验不一致时抛出乐观锁冲突异常，
HTTP 层返回 409 Conflict，前端提示"操作已被他人完成，请刷新重试"。

该设计保证即使分布式锁短暂失效，乐观锁也能兜底阻断重复审批。
在功能测试 TC-HR-06 中，通过模拟两路并发 approve 请求，验证后提交者必然收到 409。

\subsection{BEFORE\_EXECUTION 审批后防重复暂停}


当人工检查点配置为 \texttt{BEFORE\_EXECUTION} 时，
审批通过后调度器会重新执行该节点（因为节点执行前被拦截，尚未运行）。
如果不加额外控制，节点重新执行时 \texttt{checkPause()} 会再次判定满足条件，
导致工作流陷入"无限暂停"的死循环。


解决过程中，系统在 \texttt{Execution} 聚合根中维护 \texttt{reviewedNodes} 集合，
记录所有已完成审批的节点 ID。\texttt{checkPause()} 的首要判断为：

\begin{lstlisting}[style=java, caption={checkPause() 已审批节点跳过逻辑}]
if (execution.isNodeReviewed(node.getNodeId())) {
    return false;  // 已审批过，直接跳过暂停
}
\end{lstlisting}

\texttt{resume()} 时，系统将当前节点加入 \texttt{reviewedNodes} 并持久化，
从而保证节点无论被重新执行多少次，都不会再触发同一审批检查点。

\subsection{SSE 长连接稳定性与心跳保活}


工作流执行时间可能较长（例如多节点串行、LLM 推理延迟），
浏览器默认会在约 30--60 秒无数据传输后自动关闭 SSE 连接，
导致前端画布停止更新，用户误认为系统卡死。
此外，nginx 等反向代理也会因超时断开长连接。


解决过程中，系统实现了服务端主动心跳机制：
在 \texttt{SseEmitter} 建立后，后台定时任务每 15 秒推送一次
\texttt{ping} 类型的 SSE 事件（携带当前服务端时间戳），
使连接保持活跃状态直到执行完成或显式关闭。

同时，系统在 \texttt{SseEmitter} 的 \texttt{onTimeout} 和 \texttt{onError} 回调中
完成资源清理（从 emitter 池移除、取消心跳任务），
防止 emitter 泄漏积压。此方案在 SSE 推流延迟测试中验证了
100 次执行样本最大延迟 387ms，无连接中断记录。

\subsection{文档向量化阻塞上传接口}


知识库文档上传后需经历"PDF/Word 解析 → 文本分块 → 批量调用嵌入模型 → 写入 Milvus"
的完整处理链路，对于一份 50 页的 PDF 文档，
同步处理耗时可达数十秒，严重阻塞前端上传请求，造成超时或用户体验下降。


解决过程中，系统将文档处理链路设计为异步解耦模式：
上传接口仅完成文件合法性校验、MinIO 存储和 \texttt{PENDING} 状态记录后立即返回，
再通过 Spring \texttt{@Async} 触发异步处理器在独立线程中完成后续处理，
并在处理完成后回写文档状态为 \texttt{COMPLETED} 或 \texttt{FAILED}。

前端通过轮询文档列表接口观察状态变化，实现非阻塞的文档接入体验。
该设计使上传接口响应时间从同步场景的数十秒降至 500ms 以内（
仅含文件校验和 MinIO 上传耗时），显著改善了大文档场景下的用户体验。

\section{本章小结}

本章按照“实现过程—关键代码—界面展示”的思路，对智能体编排系统的核心功能实现进行了说明。用户认证模块实现了邮箱验证码、注册登录、密码找回和令牌访问控制，为后续业务操作提供身份基础；Agent 管理模块通过草稿、发布版本和历史回滚机制，将 React Flow 画布配置转化为可追溯的可执行图定义；工作流调度模块基于 \texttt{Execution} 聚合根完成 DAG 节点推进、条件分支剪枝和节点执行器分发；人工检查点模块在自动执行链路中引入暂停、审批、修改、恢复和拒绝终止机制，使系统具备关键节点人工把关能力；知识库模块通过文档上传、异步分块、向量化入库和语义检索，为 LLM 节点提供外部知识增强。最后，本章还总结了 DAG 环检测、并发审批一致性、重复暂停防护、SSE 心跳保活和文档异步处理等开发难点及解决方案。通过上述实现，系统形成了从可视化配置、自动执行、人工审核到知识增强的核心功能闭环。
