\chapter{智能体编排系统需求分析}

\section{业务需求分析}

\subsection{业务背景}

随着大语言模型能力逐步增强，企业开始尝试将大模型嵌入客服问答、
资料检索、报告生成、流程审批和业务辅助决策等场景。
成熟的低代码智能体平台以 Dify 等工具为代表，
通常围绕应用构建、可视化工作流、知识库检索增强、工具调用、应用发布与运行观测等能力展开，
帮助用户将大模型能力快速组合成可运行的业务应用。

然而，在企业内部系统落地过程中，仅具备“创建一个对话机器人”并不足以支撑严肃业务场景。
一方面，业务流程通常不是单轮问答，而是由多个步骤组成：
需要接收输入、查询知识、调用模型、判断条件、生成结果并记录执行过程；
另一方面，部分任务涉及合同内容、客户回复、审核意见或内部决策，
不能完全交由模型自动完成，需要在关键节点保留人工检查与修正入口。
因此，本系统的业务目标是提供一套面向企业业务流程的 AI Agent 编排平台，
使智能体应用能够被构建、发布、执行、审核、追踪和维护。

\subsection{核心业务需求}

围绕上述业务背景，本系统需要满足以下五类核心业务需求。

（1）智能体应用快速构建需求。
企业需要将具体业务流程抽象为可执行的智能体应用。
系统应支持通过可视化方式创建 Agent、配置节点、连接流程、保存草稿并发布版本，
使应用构建者能够在较低编码成本下完成业务流程编排。
同时，业务流程会持续迭代，系统需要支持版本发布、历史回滚与版本追溯，
避免流程调整后无法恢复到稳定版本。

（2）业务任务可触发、可观察的执行需求。
业务使用者需要像使用普通业务系统一样触发已发布的智能体应用，
并在执行过程中看到节点状态、模型输出和最终结果。
由于大模型调用和知识检索存在耗时，系统需要提供实时反馈能力，
避免用户只能在长时间等待后得到一个不可解释的最终答案。
因此，工作流执行过程必须具备状态推进、实时推送、执行日志与异常反馈能力。

（3）关键节点人工审核需求。
在企业业务场景中，模型生成结果可能影响后续决策或外部输出。
当执行链路进入高风险节点时，系统应允许业务使用者对节点输入或输出进行检查，
必要时修改中间内容，再决定通过或拒绝。
该需求要求系统能够在运行中暂停工作流、保存上下文、等待人工处理，
并在审批通过后恢复执行或在审批拒绝后终止执行。

（4）业务知识支撑需求。
企业应用不能只依赖大模型的通用知识，
还需要接入产品文档、业务制度、FAQ、操作手册和历史资料等内部知识。
系统应支持知识数据集管理、文档上传、异步解析、文本分块、向量化入库与语义检索，
并在工作流执行过程中将检索结果注入模型上下文，
从而提高回答的准确性、时效性和可追溯性。

（5）账号安全与访问控制需求。
智能体应用涉及账号身份、业务数据、审批记录和知识文档，
因此需要基础的账号注册、登录认证、访问控制、密码保护和操作留痕能力。
系统应对未认证请求进行拦截，对验证码发送、密码强度、登录失败次数等环节进行限制，
并对关键业务操作进行记录，确保平台在演示和业务验证阶段具备基本的安全性与可追溯性。

\subsection{业务参与人员}

根据上述业务需求，本系统的参与人员可归纳为两类：
应用构建者和业务使用者。
这两类人员不是从具体技术岗位直接划分而来，
而是由智能体应用从构建、发布到运行、审核的业务闭环推导得到。

（1）应用构建者。
应用构建者负责将业务流程转化为可执行的 Agent 工作流。
其主要关注 Agent 的创建、画布编排、节点参数配置、知识节点接入、版本发布和历史回滚。
该角色既可以是具备技术背景的开发人员，也可以是熟悉业务流程并具备平台配置能力的业务实施人员。

（2）业务使用者。
业务使用者负责调用已经发布的智能体应用完成具体任务。
其主要关注任务输入、执行进度、模型输出、最终结果、审批决策和历史记录。
当工作流进入人工检查点时，业务使用者还需要根据业务规则对中间结果进行确认、修正、通过或拒绝。

\subsection{Agent 管理与可视化编排需求}

Agent 管理是系统的入口能力。
应用构建者首先需要创建 Agent，录入名称、描述等基础信息，
随后进入可视化画布完成节点编排。
画布应支持配置 LLM 节点、条件节点、知识检索节点、工具节点等不同类型的执行单元，
并通过连线描述节点之间的依赖关系。

系统在保存草稿和发布版本时需要进行图结构校验。
如果图中存在环路、孤立节点或必要参数缺失，系统应拒绝保存或发布并给出明确错误提示。
发布后的 Agent 版本应作为可执行快照保存，
保证后续工作流执行时不会受到草稿继续编辑的影响。
当业务流程调整出现问题时，应用构建者应能够查看历史版本并回滚到稳定版本。

\subsection{工作流调度执行需求}

工作流执行是业务使用者直接感知系统价值的核心链路。
业务使用者选择已发布的 Agent 后输入任务内容，
系统应加载对应版本的图定义、初始化执行上下文并启动调度。
在执行过程中，系统需要按照节点依赖关系推进状态，
对无依赖冲突的就绪节点支持并行执行，
对条件分支节点支持路径选择和无效路径剪枝。

由于大模型调用和知识检索具有异步、耗时和结果不确定等特点，
系统需要通过服务端推送事件实时返回节点启动、节点完成、节点失败、
大模型流式输出和最终完成等事件。
业务使用者应能够在前端界面实时看到执行状态，
并在执行失败时根据错误信息定位问题。
系统还应持久化执行记录和关键节点日志，
为后续复盘、审计和优化提供依据。

\subsection{人工审核与流程恢复需求}

人工审核是本系统区别于普通自动化 Agent 平台的重要能力。应用构建者可以在关键节点配置人工检查点，检查点可设置在节点执行前或执行后。当工作流运行到检查点时，系统应暂停当前执行，保存执行上下文和待审核内容，并向前端推送暂停事件。

业务使用者在审批界面中查看当前节点的输入、输出和上下文信息，并根据业务要求选择直接通过、修改中间结果后通过，或拒绝并终止工作流。系统在审批提交时需要校验执行状态和版本号，防止多人同时审批造成状态覆盖，同时记录审批人、审批时间、审批结果、修改内容和备注，以满足业务追溯要求。

\subsection{知识库与检索增强需求}

知识库能力用于为智能体应用提供企业内部知识支撑。
应用构建者可以创建知识数据集并上传业务文档。
系统需要对上传文件进行类型、大小和内容合法性校验，
随后将原始文件保存至对象存储，并异步完成文档解析、文本分块、向量生成和向量入库。
文档处理完成后，系统应记录处理状态，失败时给出可追踪的错误信息。

在工作流运行时，知识检索节点根据用户输入或节点模板生成检索语句，
调用向量检索服务召回相关片段，并将检索结果注入后续大模型节点上下文。
该能力能够减少模型幻觉，使模型输出更加贴合企业资料和业务规则。
当向量检索服务不可用时，系统应记录异常并支持降级执行，
避免知识检索失败直接阻断非关键业务链路。

\subsection{用户认证与安全支撑需求}

用户认证是系统所有业务功能的访问基础。
应用构建者和业务使用者均需要先完成注册和登录，
在获得访问令牌后才能使用对应功能。
系统应支持邮箱验证码注册、账号登录、找回密码和令牌签发，
并对验证码发送频率、密码强度、登录失败次数和重复注册进行限制。

认证成功后，前端在后续请求中携带访问令牌，
后端根据令牌识别用户身份并进行访问控制。
对于 Agent 管理、工作流执行、审批决策、知识库维护等业务接口，
系统应拒绝未认证请求。

\section{非功能需求}

\subsection{性能需求}

系统应在企业级常规并发量下保持稳定响应。
Agent 管理、用户认证、知识库查询等管理类接口应具备良好的实时交互体验；
工作流执行过程中的节点状态推送与模型生成结果回显应及时、连贯，
避免出现明显卡顿或长时间无反馈。
知识文档上传应快速返回接收结果，解析与向量化处理在后台异步完成，
避免大文件处理阻塞用户界面。

\subsection{可靠性需求}

工作流执行状态、节点执行记录、审批记录、Agent 版本和文档处理状态均须持久化保存。
系统重启后，历史数据应完整可查，失败任务应保留错误信息和上下文。
在多人并发审批或重复提交场景下，
系统需要通过版本号校验或锁机制保证状态一致性，
防止同一待审批任务被重复处理。

\subsection{可扩展性需求}

系统应支持后续扩展新的节点类型、模型厂商、工具适配器、向量数据库和对象存储实现。
节点执行逻辑与底层资源访问应通过接口隔离，
避免新增节点或替换基础设施时影响已有业务主链路。
工作流图结构、节点参数和执行上下文也应保持相对通用，
为后续接入更多业务场景提供扩展空间。

\subsection{安全性需求}

所有业务接口均应基于身份认证后访问，未认证请求需被拒绝。
用户密码不得明文存储，应采用安全哈希算法保存。
邮箱验证码、登录入口和找回密码入口应具备频率控制能力，
防止恶意刷取验证码和暴力破解。
审批操作、关键执行状态和知识文档处理记录应完整留痕，
满足业务审计和问题追溯要求。
