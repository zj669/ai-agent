\section{单元测试}

系统使用 JUnit 5 + Mockito 框架，针对调度器、Agent 生命周期、
知识处理和用户认证等核心领域逻辑共编写 42 个单元测试用例，
核心领域层代码行覆盖率达 78\%，分支覆盖率达 71\%。
测试重点覆盖以下边界场景：

\begin{itemize}
  \item DAG 环检测：含环的工作流图启动时应返回“流程图存在环路”错误，工作流被阻断
  \item 就绪节点计算：给定节点状态集合，调度器应正确识别可执行节点集合
  \item 条件分支剪枝：选定分支后未选中路径上的所有后继节点应被标记为 SKIPPED
  \item 防重复暂停：节点完成审批后再次执行时应跳过暂停判断，正常完成
  \item 乐观锁校验：执行版本号不匹配时应返回 409 Conflict，阻断重复审批
  \item Agent 发布与回滚：版本快照生成、已发布版本指针更新与回滚逻辑正确
  \item 知识文档处理：文档状态在 PENDING $\rightarrow$ PROCESSING $\rightarrow$ COMPLETED / FAILED 之间正确流转
  \item 用户认证：验证码校验、密码强度校验、登录失败限制与密码哈希匹配逻辑正确
\end{itemize}

所有 42 个测试用例均已通过，无失败或跳过项。

\section{系统部署说明}

除功能和性能测试外，本文还对系统的部署方式进行说明。系统采用前后端分离和容器化部署方式，生产环境通过 Docker Compose 统一编排前端、后端和基础中间件。前端 React 应用构建后由 Nginx 容器提供静态资源访问，并将 \texttt{/api} 请求转发至后端服务；后端 Spring Boot 服务运行在 Java 21 环境中，通过容器网络访问 MySQL、Redis、MinIO 和 Milvus。该部署方式将应用服务与基础设施依赖统一纳入编排文件，降低了服务器初始化和环境迁移成本。

完成配置后，在服务器的 \texttt{docker} 目录下执行部署命令。系统支持使用已经构建好的镜像进行拉取部署，也支持在服务器上根据源码构建镜像后启动。典型部署命令如下。

\begin{lstlisting}[style=bash, caption={Docker Compose 生产部署命令}]
cp .env.prod.example .env.prod
docker compose -f docker-compose.prod.yml pull
docker compose -f docker-compose.prod.yml up -d
docker compose -f docker-compose.prod.yml ps
\end{lstlisting}

服务启动后，前端容器对外提供页面访问入口，后端容器通过健康检查接口暴露运行状态。部署完成后可通过访问前端首页、调用后端健康检查接口、查看容器状态和检查关键日志来确认系统是否启动成功。

\subsection{部署验证与运行维护}

部署验证主要包括页面访问验证、接口健康检查、中间件连通性验证和日志检查四个方面。首先访问前端首页，确认 Nginx 能够正确返回静态页面；然后访问后端健康检查接口，确认 Spring Boot 服务已启动并能够连接数据库、Redis、MinIO 和 Milvus；最后查看后端日志和容器状态，确认初始化脚本、数据卷挂载和服务依赖均正常。

\begin{lstlisting}[style=bash, caption={部署后运行状态检查命令}]
docker compose -f docker-compose.prod.yml ps
curl http://localhost:8090/actuator/health
docker compose -f docker-compose.prod.yml logs -f backend
\end{lstlisting}

运行维护方面，系统通过 Docker 数据卷持久化 MySQL、MinIO 和 Milvus 数据，常规停止命令不会清理数据卷。版本更新时，可先拉取新镜像，再执行 \texttt{docker compose -f docker-compose.prod.yml up -d} 触发服务重建；如果需要回滚，则通过指定镜像标签重新启动对应版本。由于系统包含人工审核记录、执行日志和知识库文档等重要数据，生产环境部署时还需要定期备份数据库和对象存储数据，并避免使用会删除数据卷的清理命令。
