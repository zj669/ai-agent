# 人工审核 — 业务流程与实现链路

## 一、功能概述

人工审核是工作流执行引擎中的"中断-恢复"机制。当工作流执行到配置了人工审核的节点时，引擎暂停执行，等待人工审核通过后再继续。支持两种触发阶段：

- **BEFORE_EXECUTION**：节点执行前暂停，审核人可修改输入，通过后节点才开始执行
- **AFTER_EXECUTION**：节点执行后暂停，审核人可查看并修改输出，通过后结果流转到下游

## 三、触发暂停流程

入口：`SchedulerService.checkPause(executionId, node, phase, publisher, outputs)`

```
1. node.requiresHumanReview()
   → NodeConfig.humanReviewConfig.enabled == true ?
   → false → 返回，不暂停

2. execution.isNodeReviewed(nodeId)
   → 已审核过的节点跳过（防止 resume 后重复触发）

3. config.getTriggerPhase() == phase
   → 只在配置的阶段（BEFORE / AFTER）暂停，不匹配则跳过

4. 加分布式锁 lock:exec:{executionId}

5. execution.advance(nodeId, NodeExecutionResult.paused(phase, outputs))
   → Execution 内部：
     - nodeStatuses.put(nodeId, PAUSED_FOR_REVIEW)
     - 如果有 outputs，存入 context
     - this.status = PAUSED_FOR_REVIEW
     - this.pausedNodeId = nodeId
     - this.pausedPhase = phase
     - 返回空列表（不再调度后续节点）

6. checkpointRepository.save() — 保存检查点到 Redis
7. executionRepository.update() — 持久化到 MySQL
8. publisher.publishEvent("workflow_paused", payload) — SSE 推送前端
9. humanReviewQueuePort.addToPendingQueue(executionId) — 加入 Redis Set
10. chatApplicationService.finalizeMessage() — 更新聊天消息为暂停摘要
```

## 四、审核恢复流程

入口：`POST /api/workflow/reviews/resume` → `SchedulerService.resumeExecution()`

```
1. 加分布式锁 lock:exec:{executionId}

2. 校验：
   - execution 必须存在
   - pausedNodeId 必须有值且与请求的 nodeId 匹配

3. Context Merging（区分 Phase）：
   - BEFORE_EXECUTION: edits 作为 additionalInputs（后续放入 SharedState）
   - AFTER_EXECUTION: edits 直接覆盖当前节点的 output

4. 多节点编辑 nodeEdits：遍历更新各上游节点的 output

5. 保存审核记录 HumanReviewRecord（decision = "APPROVE"）→ MySQL

6. execution.resume(nodeId, edits)
   → 状态 PAUSED_FOR_REVIEW → RUNNING
   → 清空 pausedNodeId / pausedPhase
   → reviewedNodes.add(nodeId)
   → 返回值：
     - BEFORE: [pausedNode]（需要重新执行该节点）
     - AFTER: []（已有结果，跳过执行）

7. executionRepository.update()

8. 发布 SSE 事件 "workflow_resumed"

9. humanReviewQueuePort.removeFromPendingQueue(executionId)

10. 调度下一步：
    - AFTER_EXECUTION → 构造 success result → onNodeComplete() → advance → 调度后续节点
    - BEFORE_EXECUTION → scheduleNodes(readyNodes) → 重新执行该节点
```

## 五、前端交互流程

### 待审核列表 (ReviewPage)

1. `GET /api/workflow/reviews/pending` → 从 Redis Set 获取所有待审核 executionId
2. 逐个查询 Execution，组装 PendingReviewDTO（executionId, nodeId, nodeName, triggerPhase, pausedAt, agentName）
3. 列表展示，支持"通过"、"拒绝"、"查看详情"操作

### 审核详情 (ReviewDetailPage)

1. `GET /api/workflow/reviews/{executionId}` → 获取 Execution + Context
2. 展示所有已成功的上游节点 + 当前暂停节点的输入/输出
3. 根据 triggerPhase 显示不同提示：
   - BEFORE_EXECUTION：节点尚未执行，不展示当前节点输出
   - AFTER_EXECUTION：节点已执行，展示输出且可编辑
4. 每个节点的输出可通过 Modal 编辑（JSON 格式），暂存在 `nodeEdits` state
5. 提交时将 nodeEdits 一并发送到 `/resume` 接口

## 六、整体数据流图

```
[工作流画布 — 节点配置 humanReviewConfig.enabled=true]
                    │
                    ▼
[SchedulerService 执行节点] ── checkPause() ──▶ Phase 匹配？
                    │                              │ Yes
                    │                              ▼
                    │                Execution.advance(paused)
                    │                Redis Set ← executionId
                    │                SSE → "workflow_paused"
                    │                Chat Message → 暂停摘要
                    │
                    ▼ (暂停，等待人工)
[前端 ReviewPage] ◄── GET /pending ── Redis Set
        │
        ▼
[ReviewDetailPage] ◄── GET /{id} ── Execution + Context
        │
        ├── 编辑节点输出 (nodeEdits)
        │
        ▼
[POST /resume] ──▶ SchedulerService.resumeExecution()
                    │
                    ├── Context Merge (edits / nodeEdits)
                    ├── HumanReviewRecord → MySQL (审计)
                    ├── Execution.resume() → RUNNING
                    ├── Redis Set.remove()
                    ├── SSE → "workflow_resumed"
                    │
                    └── BEFORE? → scheduleNodes(node) → 重新执行
                        AFTER?  → onNodeComplete() → 调度后续
```
