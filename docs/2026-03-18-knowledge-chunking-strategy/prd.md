# PRD：知识库上传分块策略可配置化

## 1. 背景

当前知识库文档上传链路仅支持一类固定分块配置：

- `chunkSize`
- `chunkOverlap`

这套能力可以满足基础入库，但在实际使用中有几个明显限制：

- 用户无法根据文档类型选择更合适的分块方式。
- 上传页面没有暴露分块策略配置，用户只能依赖后端默认值。
- 后端模型和数据库结构与“固定分块”强耦合，不利于后续扩展更多分块方式。
- 不同类型文档，例如制度文档、API 文档、产品说明书、长篇 Markdown，适合的切分方式并不一致。

因此，本次需求希望把知识库上传能力升级为“可选择分块策略 + 各策略独立参数配置”的模式，并同步完成前后端改造。

## 2. 问题定义

当前系统存在以下问题：

1. 上传时缺少分块方式选择能力，无法做策略化入库。
2. 文档配置落库结构过于简单，仅能表达两个通用参数。
3. 异步处理器与分块实现强绑定，后续增加新策略成本较高。
4. 前端上传体验缺少可见的高级配置区域，无法支持精细化控制。

## 3. 产品目标

本次改造目标如下：

1. 在知识库上传流程中支持选择分块策略。
2. 每种分块策略支持独立参数配置。
3. 后端以统一配置模型驱动分块，不再只依赖 `chunkSize/chunkOverlap`。
4. 保持对现有文档和旧上传调用方式的兼容。
5. 为后续新增更多分块策略预留扩展结构。

## 4. 非目标

本次不纳入范围的内容：

1. 不改造知识库检索排序策略。
2. 不改造向量表结构或重做向量集合设计。
3. 不做文档级在线重分块或重建索引入口。
4. 不在本期实现过多策略类型，首期只覆盖最有价值的两类。
5. 不强制历史文档立即迁移重灌。

## 5. 目标用户

主要用户包括：

1. 在知识库页面上传文档的业务用户。
2. 需要精细控制知识块质量的产品、运营、实施人员。
3. 需要针对不同文档类型优化 RAG 效果的开发和测试人员。

## 6. 用户故事

### 用户故事 1：固定分块上传

作为一个普通用户，我希望在上传文档时选择“固定分块”，并设置块大小和重叠大小，这样我可以延续当前行为，并按文档长度微调切分效果。

### 用户故事 2：语义分块上传

作为一个对检索质量敏感的用户，我希望在上传文档时选择“语义分块”，并调整最小块大小、最大块大小、语义阈值等参数，这样我可以减少机械切分导致的上下文割裂。

### 用户故事 3：系统兼容旧文档

作为系统维护者，我希望旧文档和旧上传调用不受影响，这样本次上线不会破坏当前知识库能力。

## 7. 需求范围

### 7.1 首期支持的分块策略

首期支持以下两种策略：

1. `FIXED`
2. `SEMANTIC`

### 7.2 FIXED 策略参数

建议参数：

- `chunkSize`
- `chunkOverlap`

### 7.3 SEMANTIC 策略参数

建议参数：

- `maxChunkSize`
- `minChunkSize`
- `similarityThreshold`
- `mergeSmallChunks`
- 可选 `chunkOverlap`

说明：

- 首期的“语义分块”允许采用规则驱动或语义近似实现，不要求一开始就必须依赖 embedding 级别的真实语义切分。
- 只要对外表现为“按语义边界优先切分”，并能支持独立参数调节，即满足本期产品要求。

## 8. 功能需求

### 8.1 前端上传页

系统需要在知识库上传区域新增“分块配置”能力：

1. 用户可选择分块策略。
2. 当选择不同策略时，展示对应参数表单。
3. 提供每个参数的默认值和说明。
4. 上传请求中带上所选策略和参数。
5. 未主动修改时，应使用系统默认配置。

### 8.2 后端上传接口

上传接口需要支持接收以下信息：

1. 基础上传参数：`datasetId`、`file`
2. 分块策略标识：`chunkStrategy`
3. 策略参数集合

同时需要满足：

1. 对旧参数调用兼容
2. 对参数进行合法性校验
3. 参数缺失时回退到默认值

### 8.3 文档配置落库

系统需要保存每个文档实际使用的分块策略和分块参数，方便：

1. 后续异步处理复用
2. 问题排查
3. 配置回显
4. 历史文档行为追踪

### 8.4 异步处理器

异步处理器需要根据文档保存的分块配置选择具体分块器：

1. `FIXED` 走固定分块实现
2. `SEMANTIC` 走语义分块实现
3. 未识别策略默认回退到 `FIXED`

## 9. 兼容性要求

系统必须满足以下兼容要求：

1. 未传 `chunkStrategy` 时，默认使用 `FIXED`。
2. 旧文档记录若无策略字段，按 `FIXED + chunkSize/chunkOverlap` 解释。
3. 旧前端上传流程在不升级页面的情况下仍可正常工作。
4. 检索链路不应因本次改造发生功能退化。

## 10. 数据要求

建议保存以下内容：

1. `chunkStrategy`
2. `chunkConfigJson`
3. 兼容期继续保留 `chunkSize` 和 `chunkOverlap`

这样既能支持新策略，又能兼容现有字段。

## 11. 交互要求

前端上传页需要做到：

1. 默认展示分块策略选择器
2. 默认策略为 `FIXED`
3. 不同策略的参数区切换时应清晰可见
4. 参数输入错误时给出明确提示
5. 上传成功后仍沿用当前“已上传，后台处理中”的反馈方式

## 12. 成功标准

满足以下条件视为本期需求完成：

1. 用户可在上传页面选择 `FIXED` 或 `SEMANTIC`。
2. 用户可配置对应策略参数并成功上传。
3. 后端可根据策略正确完成分块并入向量库。
4. 旧上传调用方式仍能继续使用。
5. 前后端关键测试通过。

## 13. 验收标准

### 验收场景 1：固定分块上传

当用户选择 `FIXED`，设置 `chunkSize=600`、`chunkOverlap=80` 上传文档后：

1. 上传接口成功返回
2. 文档配置正确落库
3. 异步处理按固定分块执行
4. 文档状态最终为 `COMPLETED`

### 验收场景 2：语义分块上传

当用户选择 `SEMANTIC` 并填写参数后上传文档：

1. 上传接口成功返回
2. 文档配置正确落库
3. 异步处理选择语义分块实现
4. 文档状态最终为 `COMPLETED`

### 验收场景 3：旧参数兼容

当调用方只传 `chunkSize` 和 `chunkOverlap`，不传 `chunkStrategy` 时：

1. 上传接口仍可成功
2. 系统按 `FIXED` 处理
3. 不出现兼容性错误

## 14. 风险与约束

1. 语义分块的技术实现复杂度高于固定分块，需要控制首版范围。
2. 数据库存储模型升级需要兼顾新旧字段兼容。
3. 前端动态表单需要与后端参数模型严格对齐。
4. 若语义分块参数设计过多，可能增加用户理解成本。

## 15. 版本建议

建议按以下节奏推进：

1. V1：支持 `FIXED` 和 `SEMANTIC` 两种策略，完成前后端打通
2. V1.1：补充分块配置展示与回显能力
3. V1.2：按使用效果决定是否继续扩展 `MARKDOWN_SECTION`、`PARAGRAPH` 等策略
