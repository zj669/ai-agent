# 业务 SOP 写作规范

## 定位

业务 SOP 是 operation-spec：给 agent 和人类执行具体业务工作的运行手册。

它不是代码说明、源码摘录、API 文档或仓库导览。代码、命令和脚本可以出现，但只能作为执行步骤的支撑事实。

## 写作原则

1. 先写业务动作，再写实现事实。
2. 先写执行顺序，再写代码入口。
3. 先写安全边界，再写可执行能力。
4. 先写验证方式，再声称任务完成。
5. 从源示例 skill 学习运行形状，不复制源工作区事实。

## 最低结构

每个业务 SOP 至少包含：

1. 触发条件。
2. 范围和必要输入。
3. 开工前读取。
4. 停止条件和确认包。
5. 操作步骤。
6. 安全边界。
7. 代码定位或脚本使用规则。
8. 验证步骤。
9. 输出格式。
10. 知识回写。

## 操作步骤模板

操作步骤必须覆盖完整闭环：

1. 定界：确认业务动作、目标对象、时间范围、环境和影响面。
2. 只读调查：先收集证据，不修改代码、数据或生产状态。
3. 操作计划：区分只读、代码变更、部署、数据库写入、账号/权益、财务和通知动作。
4. 确认：高风险动作先准备确认包，得到用户明确同意后再执行。
5. 执行：按最小影响范围执行，优先复用已有流程和脚本。
6. 验证：覆盖直接验证、依赖链验证和端到端验证；未验证项必须说明。
7. 回写：把长期有效经验写到最窄归属文件。

## 代码内容边界

允许写：

- 真实代码入口路径。
- 需要搜索的函数、路由、service、job、model、脚本名。
- 命令签名、脚本参数和环境变量名。
- 极短示例，且必须服务于某个步骤。

禁止写：

- 大段源码复制。
- 按文件逐段解释实现但没有行动步骤。
- 只有 API、函数实现或类结构，没有触发、确认、验证、回写。
- 把仓库、服务、文件或函数提升为一等业务路由，除非用户明确要求项目导向 skill。
- 从源示例 skill 复制具体业务、域名、服务、数据库、事故或部署规则。

## 好坏判断

好的 SOP 能回答：

- 用户说什么时应该进入这个 SOP？
- 开工前读什么？
- 需要用户给什么输入？
- 哪些动作能直接做，哪些必须停下来确认？
- 调查、执行、验证和回写的顺序是什么？
- 如果涉及代码，应该从哪里定位真实实现？
- 如何证明结果完成？

坏的 SOP 通常表现为：

- 只有仓库或文件列表。
- 只有代码片段或函数说明。
- 没有停止条件。
- 没有验证标准。
- 没有知识回写位置。
- 把源示例工作区事实当成目标事实。
