# 业务分层工作区 Skill 验证清单

## 结构

- 根 `SKILL.md` 存在。
- `.env.example` 存在，包含目标工作区实际使用技术栈的占位符变量，不含真实密钥。
- 根 `SKILL.md` 有包含 `name` 和 `description` 的 YAML frontmatter。
- Frontmatter description 说明何时使用该 skill，并包含可能的业务触发措辞。
- 必要目录存在，或被明确说明为有意省略：
  - `logs/`
  - `references/`
  - `references/incidents/`
  - `scripts/`
  - `agents/`
  - `evals/`
- 被引用的 `template/` 文件存在。
- 被引用的 `references/` 文件存在。
- `references/required-data.md` 存在，并在初始化或数据完整性审计前使用。

## 业务分层完整性

根 `SKILL.md` 应包含：

- 身份和边界；
- 强制 first-read 或最近日志规则；
- 路径约定；
- 业务域地图；
- 歧义业务术语的边界说明；
- 全局安全规则；
- 如果可能触及数据库，则包含数据库安全；
- 代码搜索和复用规则；
- 如果可能部署，则包含部署规则；
- 如果存在 Web 前端，则包含前端/浏览器验证规则；
- 验证标准；
- 输出规则；
- 知识回写规则；
- `Meet -> See` 业务路由表。

## 必要数据

- 已收集 `references/required-data.md` 中的初始化最小字段，或明确列为缺失。
- 相关时，`references/required-data.md` 中的运维数据字段已体现在 `.env.example`。
- 支撑仓库/服务/域名/数据库附着到业务域，不提升为一等路由归属。
- 在任何 SOP 描述执行能力前，安全数据已经存在。
- 缺失必要数据被报告，而不是从猜测或源示例填充。

## 路由完整性

- 每个业务域地图路由都指向已存在文件或明确的缺失占位。
- 每个 `Meet -> See` 路由都指向已存在文件或明确的缺失占位。
- 域 `index.md` 文件不重复根级全局规则。
- 具体 SOP 不声称拥有整个工作区。
- 仓库、服务、域名、数据库和项目名是触发提示或支撑事实，不是默认一等路由归属。
- 歧义业务术语有路由优先级说明。
- 高风险路由指向有安全边界的 SOP。

## SOP 步骤质量

- 生成或维护业务 SOP 前已应用 `references/sop-authoring-spec.md`。
- 业务 SOP 是具体步骤型运行手册，而不是代码合集。
- 每个业务 SOP 都有触发条件、必要输入、开工前读取、操作步骤、停止条件、安全边界、验证、输出格式和知识回写。
- 操作步骤说明调查、计划、执行、验证、回写的顺序。
- 代码入口、函数名、脚本名和命令只作为支撑事实出现。
- 不存在大段源码复制。
- 不存在只有 API、函数实现或文件逐段解释而没有行动步骤的 SOP。
- 不把仓库、服务、文件或函数提升为一等业务路由，除非用户明确要求项目导向 skill。
- 从源示例 skill 学习结构时，只迁移运行形状，不迁移源业务事实。

## 安全

- 生产服务器编辑有护栏。
- 数据库写入要求确认和验证。
- 不可逆变化被明确指出。
- 凭据和 token 处理政策明确。
- 适用时，账号池和财务工作流被视为高风险。
- 适用时，前端变更要求浏览器验证。
- 未知安全边界标记为 `requires-confirmation`。

## 可移植性

- 模板使用占位符，而不是真实企业 secrets 或只适用于某台机器的绝对路径。
- 生成后的目标 skill 优先使用相对工作区根目录的路径。
- 脚本在需要可移植性时，通过环境变量定位 skill / 工作区根目录。
- 真实凭据、token、webhook URL、账号 session 和 refresh token 不放入通用模板。
- `.env.example` 只包含占位符。
- 本 skill 不生成真实 `.env` 文件。


## 泄漏扫描

最终确定生成后的目标 skill 前，从源示例工作区定义禁用词集合，并扫描生成输出。

至少包括：

- 源工作区名称；
- 源仓库名称；
- 源服务名称；
- 源域名、URL、端口和 IP；
- 源数据库名和表名；
- 源专属业务标签。

任何命中都必须解释为：

- `confirmed-target-fact`: 已在目标工作区独立发现；
- `allowed-migration-fact`: 因为这是同一工作区或原地迁移，所以有意保留；
- `leak`: 意外迁移，必须移除。

## 维护

- 新业务域有域索引文件。
- 新工作流有具体业务 SOP 或明确缺失占位。
- 交叉链接已维护。
- 废弃路由被标记，而不是静默删除。
- 经验教训写入最窄相关文件。
- 必要时，最近用户纠正同时体现在日志和可复用规则中。
